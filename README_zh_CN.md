# JavaScript 运行时的混合 Arena + GC 内存模型

**核心思路**：让 GC 通过 SPI 学习业务上下文。与其让 GC 猜测哪些对象是短命的，不如暴露一套最小化的 SPI，允许宿主运行时将请求级别的生命周期边界直接告知引擎。

---

## 问题所在

V8 完全不知道你的请求边界在哪里。为请求 #4821 分配的 300 个对象将在 12ms 后全部死亡——但 V8 不知道这件事。它反复追踪这些对象，按自己的节奏回收它们，导致 p99 延迟毛刺，无论怎么调优都无法真正消除。

---

## 模型

当 HTTP 请求到来时，Node 创建一个绑定到其异步上下文的 **Arena**。所有对象通过指针碰撞分配落入其中。请求结束时，整个 Arena 通过一次 `free()` 调用释放——无追踪，无标记，无清扫。

GC 暴露三个 SPI 调用：

```c
Arena *createArena();
void releaseArena(Arena *arena);
void setContextArena(Arena *arena);
```

> Arena的初始容量较小（如 1KB），按需通过 chunk 链表动态扩容，上限仅受整个 Arena 堆总容量约束。

Node 的 HTTP 层调用这些接口。应用代码什么都不调用。`express`、`fastify`、`koa`——所有现有代码无需任何修改即可受益。

对象分配规则：
- **Arena 上下文激活时** → 分配到 Arena
- **无 Arena 上下文**（非 Node 环境、长连接）或大对象 → 标准 GC 堆

---

## 不变量

> Arena 对象可以引用 GC 对象。GC 对象绝对不能引用 Arena 对象。

通过在 V8 现有写屏障上增加一个位掩码检查来强制执行：

```
写入时：target[field] = value
  如果 address_of(value) 在 Arena 地址范围内
  且 address_of(target) 不在 Arena 地址范围内：
    promote(value)
```

---

## 晋升：真实的成本

1. 当 GC 对象持有对 Arena 对象的引用时，`promote()` 被触发。它对逃逸子图执行 DFS 遍历，将所有可达的 Arena 对象移入 GC 堆并更新它们的指针。

2. 从 10,000 个对象中晋升 100 个，仍然需要扫描整个 Arena，以修正剩余 9,900 个可能引用了被晋升对象的对象中的指针*（是否引入 Remembered Set 来优化晋升时的指针修正成本，需实测验证，而非先验判断）*。

**为什么没有听起来那么糟**：

每个请求的 Arena 是一座孤岛。设计上不存在跨 Arena 的引用——请求 A 的对象永远不会引用请求 B 的对象。指针修正的范围被严格限定在单个请求的 Arena 内，而不是整个堆或整个年轻代。

在拥有数万个并发 Arena 的高并发场景下，每个 Arena 的晋升操作完全独立。不存在全局的 stop-the-world 停顿——修正工作分散在各个请求中，而不是汇聚成一次全局阻塞事件。

真正的问题是一个实证问题：**在典型的 HTTP 工作负载中，对象逃逸的频率有多高，逃逸子图有多大？** 如果逃逸是罕见的——在结构良好的应用中通常如此——Arena 以零追踪成本释放绝大多数对象，晋升只处理例外情况。

---

## 这是分代 GC——重新设计的版本

经典分代 GC 在没有业务上下文的情况下应用分代假说：频繁回收一个小苗圃，晋升存活者。年轻代之所以小，是因为 GC 必须急切地回收以释放空间。

本提案给了 GC 它从未拥有过的东西：**来自业务层的显式生命周期边界**。结果是一个大小恰好等于请求的年轻代，在恰当的时刻被精确地回收一次，在常见情况下追踪开销为零。

---

## 彩蛋：异步 JSON 序列化

Arena 根对象的响应是请求级作用域的，不与并发请求共享。这使得在 **libuv 线程池线程**上序列化它们变得安全，而不是阻塞事件循环。

如果 I/O 线程在遍历过程中遇到 GC 对象引用，就写入一个占位符。主线程事后只补全那些碎片。在最常见的情况下——完全由 Arena 根对象构成的响应——`JSON.stringify` 完全在主线程之外完成。

*（推测性方向——线程安全细节需要仔细分析。）*

---

## 这不是什么

- 不是替代 GC。GC 处理所有逃逸的对象。
- 不是开发者 API。应用代码零改动。
- 不适用于 WebSocket、SSE 或长连接——这些使用标准 GC 堆。

---

## 难点所在

- 写屏障必须覆盖所有引用写入路径：JIT 编译代码、解释器、原生扩展、WeakMap。
- 晋升 DFS 必须相对于 JS 执行是原子的。
- 双堆需要与 V8 现有分配器进行谨慎的虚拟内存布局协调。
- DevTools 内存分析器和堆快照需要更新。

这些都不是无解的问题。但都需要 V8 团队的参与。
