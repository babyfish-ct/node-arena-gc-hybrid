# 如果 Node.js 能实现零 GC 请求处理，而你不需要改一行代码，会怎样？

## 一个将混合 Arena + GC 内存模型内置于 JavaScript 运行时的提案

---

## 没人认真谈的问题

Node.js 每天在全球生产系统中处理数百万请求。然而，GC 停顿问题从未真正消失。

你懂那个症状：**p99 延迟毛刺**。不是 p50，不是 p95——而是那个残酷的 p99 尾延迟，你的 SLA 要求你解决，但你永远无法彻底消除。罪魁祸首几乎总是同一个：V8 的垃圾回收器在最糟糕的时刻启动，暂停你的事件循环，冻结你正在处理的请求。

标准建议？调 `--max-old-space-size`。减少内存分配。用对象池。用 `--trace-gc` 做性能分析。

这些都是创可贴。它们没有触及根本原因。

**根本原因是：V8 根本不知道你的请求边界在哪里。**

它不知道你刚刚为请求 #4821 分配的 300 个对象，在 12 毫秒后响应发出时就全部死亡了。它只看到一堆满是对象的堆，然后按自己的节奏运行 mark-and-sweep 算法，毫无区分。

如果运行时*知道*这件事呢？如果它能利用这个知识，让你几乎所有的请求对象分配和释放接近零成本呢？

---

## 核心思路：请求级 Arena 分配，开发者完全不可见

核心提案，简单说：

当一个短生命周期请求（如 HTTP 请求/响应周期）进入 Node.js 运行时，运行时自动创建一个 Arena——一块连续内存区域——绑定到该请求的异步上下文。 在这个异步作用域内分配的所有 JavaScript 对象都放在这个 Arena 里。如果请求**足够幸运**，请求结束时所有 JavaScript 对象都不再有用，整个 Arena 一次性释放。无需 GC。

这里先讨论幸运的场景，不幸运的场景稍后再谈。

---

## 架构：双堆 + 地址掩码写屏障

**核心解决方案**：与其让 GC 去猜测哪一小部分对象是短命对象，不如让 GC 提供 SPI——让 Node 层利用这套 SPI 告诉 GC 在真实业务场景中实际存在的事实：大部分对象都是短命对象。

**最终目标**：业务代码对这一切完全无感。

### 两个堆，而非一个

基本架构选择是**双堆模型**：

- **GC 堆**：标准的 V8 托管堆。所有长生命周期对象在这里。GC 照常追踪、标记、清扫这个堆。
- **Arena 堆**：一块独立的内存区域，在 GC 堆的地址范围之外，专用于 Arena 分配的对象。

**堆归属完全由地址范围决定**。一个对象是 Arena 对象，当且仅当其地址落在 Arena 堆的地址范围内。无需修改对象头部，无需破坏 V8 精心设计的紧凑对象格式。

GC 暴露以下 SPI：

```c
Arena *createArena();
void releaseArena(Arena *arena);
void setContextArena(Arena *arena);
```

- **`createArena`**：从 Arena 堆分配一个 Arena。初始体积小（如 1KB），但支持通过 chunk 链动态扩容。
- **`releaseArena`**：释放 Arena，其中寄宿的所有对象自动死亡。
- **`setContextArena`**：设置当前主线程的 Arena 上下文。

该机制按如下方式工作

-   当 Node 开始处理一个请求时，它会创建一个 Arena。一旦所有异步流程完成——无论成功还是失败——该 Arena 就会被释放。在整个过程中，Arena 上下文由关联的 AsyncContext 进行追踪。

-   当主线程获得控制权时，它会设置 Arena 上下文。在让出控制权之前，它会将其设回 NULL。

-   每当主线程创建一个 JavaScript 对象时，遵循以下规则：

    -   大对象完全忽略 Arena 上下文，直接在 GC 堆上分配——从一开始就作为 GC 对象诞生。

    -   如果当前 Arena 上下文为 NULL，对象直接在 GC 堆上分配——作为 GC 对象诞生。这涵盖了所有没有人调用 SPI 的场景，尤其是非 Node.js 环境。

    -   如果当前 Arena 上下文非 NULL，则从当前 Arena 中分配内存，产生的是一个 Arena 对象，而非 GC 对象。

### 不变量

前面讨论的场景——请求结束时 Arena 内所有 JavaScript 对象都不再有用——是非常幸运的情况。然而现实并非总是如此理想。

整个模型由一个不变量约束：

> **Arena 对象可以持有对 GC 对象的引用。GC 对象绝对不能持有对 Arena 对象的引用。**

Arena 对象持有指向 GC 堆的引用是完全安全的——GC 对象的生命周期长于 Arena，所以那些引用始终有效。危险的方向是反过来：GC 对象引用 Arena 对象，在 Arena 被释放的瞬间就会变成悬空指针。

### 基于地址掩码的写屏障

V8 已经为分代 GC 维护了写屏障。本提案用一个极其廉价的检查来扩展写屏障：

```
每次引用写入：target[field] = value

  如果 address_of(value) 在 Arena 堆地址范围内：
    如果 address_of(target) 不在 Arena 堆地址范围内：
      // GC 对象试图持有 Arena 引用 → 触发晋升
      promote(value)
```

地址范围检查实现为**位掩码操作**——一次 AND，一次比较。在本来就对每次引用写入都运行的写屏障上下文中，这个开销实际上可以忽略不计。无哈希查找，无元数据表，无对象头部读取。

当 `promote(value)` 被触发时，从 `value` 出发做深度优先遍历，将所有可达的 Arena 对象晋升到 GC 堆，确保在写入完成之前不变量得以恢复。

写屏障至关重要。这套 `promote` 机制是防止 Arena 提前释放导致悬空指针的关键所在。

### Arena 标记分配

当一个短生命周期异步上下文被创建（如 HTTP 请求），Node 在 Arena 堆中创建一个 Arena 并通过 `AsyncContext` 与该上下文关联。该上下文内所有分配使用 Arena 内的指针碰撞分配。当没有活跃的 Arena 上下文时——包括所有非 Node 环境和长连接场景——分配降级到标准 GC 堆。

### 请求完成 = Arena 释放

当异步上下文结束，Arena 一次性释放——一次 `free()` 调用，可能同时释放数千个对象。无追踪，无标记，无清扫。

## 这是分代 GC——为业务负载重新设计

值得明确说清楚这个模型在更深层次上究竟是什么。

**这是分代垃圾回收的一个变种。**

经典分代 GC 建立在*分代假说*之上：大多数对象死得早。年轻代是一块小的、快速回收的区域；经历足够多 GC 轮次存活下来的对象被晋升到老年代。

本提案采用同样的假说，并用运行时从未拥有过的**业务领域知识**来应用它：

- 经典年轻代：体积小（几 MB），每隔几毫秒回收一次，对象通过熬过 GC 轮次来存活
- Arena 代：**大小随请求需求而定**，恰好回收一次（在请求结束时），对象通过显式晋升来存活

关键洞察在于：对于高吞吐量 HTTP 工作负载，*请求边界*是一个比任何通用 GC 周期能推断出的更精确、更有意义的生命周期边界。与其让 GC 通过观察分配压力来猜测哪些对象是短生命周期的，不如直接告诉运行时生命周期边界在哪里。

**Arena 是一个与业务负载实际对象生命周期分布完美匹配的年轻代。** 它不是一个小的、急于回收的苗圃——它是一个请求大小的代，在恰当的时刻被精确地回收一次，追踪开销为零。

## 为什么可行：大多数对象其实很"无聊"

让这个方案可行的洞察简单得令人尴尬：

**典型 Web 请求中，绝大多数对象的生命周期恰好是一个请求。**

你的 ORM 查询结果？响应时死亡。你的 DTO？响应时死亡。你的中间件上下文对象？响应时死亡。你解析的 JSON body？响应时死亡。

在一个结构良好的 Node.js 应用里，可能 95% 以上的分配对象从不逃逸出创建它们的请求。它们一起诞生、使用、消亡。但今天的 GC 不知道这一点——所以它把它们和长生命周期对象一样对待，反复追踪它们，最终才回收。

有了请求级 Arena，这些对象的代价几乎为零：

- **分配**：指针碰撞，接近零成本
- **释放**：请求结束时整个 Arena 一次释放，每个对象的平摊成本接近零
- **GC 介入**：零。GC 根本看不到这些对象

真正逃逸的少数对象——放入内存缓存的条目、单例服务、WebSocket 状态——在运行时检测到跨边界写入时被自动晋升，GC 正常处理它们。

---

## 不变量

模型维护两个关键不变量：

1. **GC 到 Arena 无跨堆引用**：晋升之后，任何 GC 管理的对象都不得引用 Arena 对象。写屏障通过在检测到逃逸时晋升整个可达子图来强制执行这一点。

2. **Arena 对象是请求级作用域的**：Arena 内所有对象共享恰好一个生命周期——创建它们的请求。Arena 对象在未被显式晋升到 GC 堆的情况下，不能存活超过其请求。

这两个不变量保证了请求完成时，整个 Arena 可以一次性安全释放，不存在悬空指针的风险。

---

## 开发者完全无感知

这**不是**面向应用开发者的新分配器 API。这**不是** pragma 或注解。这**不是**新的语言特性。

你这样写：

```javascript
app.get('/users/:id', async (req, res) => {
  const user = await db.findUser(req.params.id);  // Arena 分配
  const dto = transformUser(user);                 // Arena 分配
  const response = buildResponse(dto);             // Arena 分配
  res.json(response);
  // 请求结束 → Arena 释放 → 以上全部瞬间消失
});
```

什么都没变。没有 `arena.alloc()`。没有 `defer arena.free()`。没有生命周期注解。你的现有代码、你的现有第三方库、你的现有 ORM——全部静默受益。`express`、`fastify`、`koa`——它们都直接可用。因为这完全在运行时内部。

## `promote()` 的场景：当你真的需要全局状态

```javascript
const cache = new Map(); // GC 管理（模块作用域，无 Arena）

app.get('/config', async (req, res) => {
  const config = await loadConfig(); // 初始分配在 Arena
  cache.set('config', config);       // 写屏障触发！
                                     // address_of(config) → Arena 堆地址范围
                                     // address_of(cache 条目) → GC 堆地址范围
                                     // → 触发 promote(config)
                                     // config 及其整个对象图移入 GC 堆
  res.json(config);
});
```

开发者什么都没写。写屏障检测到地址范围违规并自动晋升了对象图。`config` 现在由 GC 管理，存活直到从 `cache` 中删除。本次请求的其他所有对象仍随 Arena 消亡。

## 彩蛋：通过 libuv 线程池实现异步 JSON 序列化

在这个模型下，还有一个额外的优化变得可能——它解决了 Node.js 长期存在的一个痛点：**`JSON.stringify` 阻塞主线程**。

在本提案下，运行时在序列化时知道被序列化的根对象是 GC 对象还是 Arena 对象。这个区别打开了一条新的优化路径：

**情况 A：GC 根对象** — 对象图可能包含长生命周期引用和共享状态。序列化照旧在主线程进行。无变化。

**情况 B：Arena 根对象** — 对象图是请求级作用域的。从构造上讲，它不与任何其他并发请求共享。这使得在主线程之外序列化它变得安全。

当序列化根是 Arena 对象时，运行时将序列化任务分发给 **libuv 的线程池**。I/O 线程遍历对象图并序列化它。如果遇到对 GC 对象的引用（这是合法的——Arena 对象可以引用 GC 对象），就在输出缓冲区写入一个**占位符**并记录该引用。

I/O 线程完成后，控制权返回主线程。如果存在占位符，主线程只序列化那些 GC 根的碎片并拼接到输出中。如果没有占位符，结果立即就绪。

在最常见的情况下——一个完全由 Arena 根对象构成、不含 GC 引用的响应对象——`JSON.stringify` 完全在主线程之外完成，**主线程阻塞时间为零**。

这对大型 API 响应尤其重要：当前序列化一个 500KB 的响应对象会在整个序列化期间占用事件循环。在这个模型下，这部分工作移入后台线程，事件循环得以保持空闲，随时处理下一个请求。

> **注意**：这个优化作为值得探索的方向提出，而非完整规范的设计。Arena 生命周期、线程安全与并发序列化过程中 GC 对象引用之间的交互，需要仔细分析。这里列出是为了展示双堆模型在降低 GC 压力之外更广泛的潜力。

## 边界情况：何时不使用 Arena

并非所有异步上下文都适合 Arena 分配：

- **WebSocket 和 SSE**：可能持续数分钟乃至数小时的长连接，绕过 Arena 分配，直接分配到 GC 堆。运行时自动识别这些持久上下文。
- **流式响应**：运行时可能为每个 chunk 而非每个请求创建新的 Arena。
- **嵌套请求上下文**：当一个请求派生另一个请求（如内部 `fetch`），子请求默认继承父请求的 Arena。
- **非 Node 环境**：CLI 工具、构建脚本、长期运行的后台 worker——这些都不设置 `Arena*`，行为与今天完全相同。

决策在上下文创建时根据异步资源类型做出。HTTP 请求/响应是主要受益者。其他一切保持不变。

## 与现有方案的对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **对象池** | 复用对象，减少分配 | 需要开发者付出努力；只对特定类型有效 |
| **减少分配** | 降低 GC 压力 | 逆 JavaScript 本性；让代码变丑 |
| **Bun / JavaScriptCore** | 更快的 I/O，原生代码 | 相同的 GC 模型；没有解决根本问题 |
| **Rust/Go 重写** | 零 GC，可预测延迟 | 丢弃生态和团队知识 |
| **本提案** | 开发者零努力；所有现有代码受益；解决根本原因 | 需要 V8 实现工作 |

## 难点（诚实说）

这对 V8 来说不是微小改动。挑战是真实的：

- **双堆地址空间管理**：保留和管理独立的 Arena 堆地址范围，需要与操作系统分配器和 V8 现有内存管理协调。地址掩码方案避免了对象头部改动，但需要谨慎的虚拟内存布局设计。

- **写屏障覆盖完整性**：基于地址范围的写屏障必须覆盖所有引用写入路径——JIT 编译代码、解释器、原生扩展、WeakMap、V8 内部结构。漏掉任何一条路径都是内存安全 bug。

- **晋升的原子性**：晋升对象图的 DFS 遍历必须相对于 JavaScript 执行是原子的。不能有任何 GC 或其他操作观察到部分晋升的对象图。

- **工具链**：Chrome DevTools 内存分析器、堆快照和分配追踪器需要理解双堆模型。

这些都不是无解的问题。但需要 V8 团队的参与——这是 RFC 级别的提案，不是周末项目。

## 哲学层面的意义

系统编程世界几十年前就知道 Arena 分配。游戏引擎用它。编译器用它。高性能 C++ 服务器用它。Apache 的 `apr_pool_t` 从 2000 年代初就在做请求级 Arena 分配了。

传统观点认为托管语言无法拥有这个，因为"你不控制分配"。

但这只在你把运行时视为语言坐在上面的黑盒时才成立。如果运行时*就是*实现本身——如果 Node.js 选择实现这个——那么托管语言就同时拥有了 GC 的人机工程学*和* Arena 分配的性能特性，在常见场景下两者兼得。

**GC 不是敌人。GC 做它不需要做的工作，才是敌人。**

你的大多数对象不需要 GC。它们需要一个 bump allocator 和请求结束时的一次 `free()`。让 GC 专注于真正需要它的对象。

---

## 结论

这个提案不是要替代 GC。而是通过给运行时赋予**请求边界的语义感知**——这种感知本来就可以有，只是从未被利用——让 GC 做更少的工作，并以此实现一种完美适配高吞吐量 HTTP 工作负载实际对象生命周期分布的分代 GC。

结果：

- **Arena 对象**：约 95% 的请求分配。近零分配成本。近零释放成本。GC 从不接触它们。
- **GC 对象**：约 5% 真正逃逸的。正常处理。
- **开发者体验**：不变。零新 API 需要学习。零迁移成本。所有现有代码受益。
- **异步 JSON 序列化**：一条将大型响应序列化彻底移出主线程的可能路径。

**最好的性能优化，是你的用户根本不需要思考的那种。**
